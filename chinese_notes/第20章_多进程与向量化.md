# 第 20 章 多进程与向量化

## 20.1 写作动机

ML 涉及数学中最计算密集的任务之一——本章讲如何用 Python 的并行/向量化把日级运行时间降低到分钟级。

## 20.2 向量化示例

向量化 = 用 numpy 数组操作代替 Python for-loop。numpy 在 C 层面执行循环,可达 100x-1000x 加速:

```python
# 慢
for i in range(len(a)):
    out[i] = (a[i] - b[i]) ** 2

# 快
out = (a - b) ** 2
```

**经验法则:** 任何能改写为数组运算的循环都应该改写。

## 20.3 单线程 vs. 多线程 vs. 多进程

| 方式 | GIL 限制? | 适合什么 | 内存 |
|------|----------|----------|------|
| 单线程 | — | 简单逻辑、小数据 | 共享 |
| 多线程 (threading) | 是 | I/O 密集 | 共享 |
| 多进程 (multiprocessing) | 否 | CPU 密集 | 独立(需 pickle) |
| Cython/Numba | 否 | 数值核心循环 | 共享 |

**Python ML 几乎都用 multiprocessing**,因为 GIL 让多线程在数值计算上没用。

## 20.4 原子 (Atom) 与分子 (Molecule)

把任务拆成不可再分的"原子"(如一行 / 一个时间戳),把多个原子组成"分子"(每个进程处理一个分子)。

### 20.4.1 线性分割 (linParts)
对相互独立的原子做均匀切分。

### 20.4.2 双层嵌套分割 (nestedParts)
对**上三角矩阵型**任务(如 $T \times T$ 协方差中只算上三角),需要根据 $i \cdot (T - i / 2)$ 切分,使每个分子工作量近似相等。

## 20.5 多进程引擎

### 20.5.1 准备 jobs
每个 job 是一个 dict,包含 `func` 和它的参数。

### 20.5.2 异步调用
用 `multiprocessing.Pool.imap_unordered` 不阻塞,先返回的先处理。

### 20.5.3 解包回调
统一解包 `func(**kwargs)`。

### 20.5.4 Pickle/Unpickle
所有传给子进程的对象都必须可序列化(避免局部 lambda、closures)。

### 20.5.5 输出归约
合并子进程结果。如果是 DataFrame → `pd.concat`;Series → 同样。

## 20.8 本章核心要点

1. **能向量化就向量化**——比并行更快、更简单。
2. ML 用 **multiprocessing** 不用 threading(GIL)。
3. 对上三角型任务用 **nestedParts** 而不是 linParts,负载更均衡。
4. 用 **mpPandasObj** 把 DataFrame 操作变成"自动并行"。
5. 大规模研究计划 IO/CPU/RAM 时,先估算每个原子的耗时和内存。
